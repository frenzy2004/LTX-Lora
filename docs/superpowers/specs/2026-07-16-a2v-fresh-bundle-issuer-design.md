# Fresh A2V Bundle Issuer Design

## Goal

Create one production-safe, offline-only command that prepares a new immutable
Fal A2V training bundle from an already reviewed private source run.  It must
preserve the approved 17 groups and their 12-training/5-holdout split, bind two
heldout provider-validation pairs, and leave the existing paid executor as the
only route that can submit a chargeable request.

## Decision

Use a new `a2v_refresh` module and a narrow `scripts/refresh_a2v_run.py` CLI.
The command takes a canonical private root, pilot ID, source execution ID, its
expected source bundle ID, and a new opaque target execution ID (never arbitrary
source or destination paths),
already-captured private price evidence, an already-recorded private standing
authorization, an explicit two-holdout prompt map, a supplied repository
commit, and fixed UTC creation/expiry timestamps.  It creates a new canonical
run under the same pilot and ledger.

The command performs no provider import, network access, credential lookup,
price capture, receipt issuance, upload, or queue submission.  Existing
commands remain responsible for fresh price capture, standing authorization,
receipt issuance, preflight, and the one-shot paid request.

## Alternatives considered

1. Reissue the expired source bundle in place. Rejected: the ledger makes a
   bundle/execution replay unsafe and it would weaken provenance.
2. Manually edit the source JSON files. Rejected: it cannot prove the archive,
   dataset manifest, execution configuration, selection, and root manifest are
   mutually bound.
3. Build a fresh canonical run from the approved source. Chosen: it keeps the
   original media untouched and produces one independently verifiable target.

## Inputs and boundaries

The source and target are resolved strictly through the configured private root
as `<root>/pilots/<pilot>/runs/<execution>`.  The target run must not exist.
The source execution configuration identifies the pilot and ledger; its fixed
A2V settings are validated before reuse.  The source bundle is verified
statically against its expected bundle ID and root artifact hashes without
treating its old temporal expiry as execution authority.  Source candidate
files, structural report, and quality attestation are revalidated and the
report must reproduce byte-for-byte before anything is copied.

Only the exact four files of every accepted source group are copied to the
target as independent regular files.  The accepted split is preserved exactly:
12 training groups and 5 holdouts.  The copied quality attestation remains
canonical and must validate against the newly reproduced structural report.
There is no automatic rejection, crop, resampling, or media mutation.

The command accepts only existing canonical price-evidence, standing
authorization, and validation-prompt-map files.  It copies and validates the
price evidence and standing authorization into the target control directory.
Their expiries must outlast the target execution expiry.  The prompt map must
contain exactly two canonical prompts keyed by distinct accepted holdout IDs in
canonical order; it is not inferred or silently generated by the issuer.

## Target artifacts and ordering

1. Secure-copy and verify the source inputs into a unique private staging
   directory.  Create `candidates/`, `control/`, `validation/`, and `bundle/`
   there using owner-only permissions; publish the fully verified staging tree
   atomically to the non-existing target only at the end.
2. Revalidate the source data and copy the approved files plus canonical
   structural report and quality attestation.
3. Write a generic private plan that records the fixed one-run, $6.0000
   training ceiling and no-third-party-lip-sync boundary without personal
   names, credentials, raw media paths, or URLs.
4. Build the deterministic training archive from the copied 12 training groups
   and write a dataset manifest that covers the copied train and holdout groups.
5. Write an exact `a2v-execution-config-v2` configuration.  It has the fixed
   A2V endpoint, rank 32, 1,000 steps, `0.0002` learning rate, 89 frames at
   24 fps, high/9:16 resolution, disabled auto-scale/scene splitting/debug
   output, enabled audio normalization/pitch preservation, and fixed
   $6.0000/$1.2500/$12.0000 ceilings.  Its digest bindings point to the newly
   built archive/dataset and copied policy/price artifacts.
6. Build exactly two validation entries from the prompt map and the copied
   start-image/audio pairs.  The selection binds the fresh structural report
   and execution configuration digests.
7. Build the root manifest from all required artifact digests and return only
   a sanitized public result: target execution ID and bundle ID.

All writes are create-only.  Any failure removes only the ephemeral staging
directory; the source run, ledger, price evidence, policy source, prompt map,
and any pre-existing target are never modified.  The output is byte-identical
when every explicit input, including IDs, timestamps, repository commit, and
prompt map, is identical.

## Safety and error handling

The issuer rejects aliases, links, noncanonical paths, source/target identity
mismatches, stale or malformed policy/price artifacts, invalid source config,
wrong group counts/splits, unexpected candidate files, any source digest drift,
and invalid prompt/config/root schemas.  It never exposes source paths, raw
identifiers, media URLs, or credentials in normal CLI output.

An execution receipt is deliberately out of scope.  It must be created only
after a policy-only preflight validates the resulting bundle and current ledger
balance.  The one-shot executor then requires the receipt-bound preflight before
it can reserve or submit the $6.0000 training request.

## Test plan

- Fresh issuance creates a canonical target with 17 groups, 12/5 split, exact
  archive membership, valid config, valid two-item provider selection, and a
  recomputable bundle ID.
- Source files remain byte-identical and source directories are never written.
- Target creation rejects existing targets, aliases, noncanonical IDs, a source
  run from another pilot, stale/malformed price, policy, or prompt-map files,
  source structural drift, and a source bundle hash mismatch.
- Selection uses two distinct accepted holdouts in canonical order and has no
  train-group reference.
- The CLI exposes no Fal key, endpoint override, budget override, media URL, or
  paid-execution flag; tests prove no provider module is imported and a freshly
  issued target passes policy-only preflight once its ledger is supplied.
